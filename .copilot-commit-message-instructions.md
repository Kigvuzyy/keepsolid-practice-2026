# Commit Message Generation Standard (Copilot Prompt)

You are generating a single Git commit message based on the diff / staged changes.

## OUTPUT FORMAT (STRICT)

Produce the commit message in this exact structure:

1. **Subject line** (one line)
2. **Blank line**
3. **Body** as a bullet list (each line starts with "- "), one item per meaningful change (optional but recommended)

No extra sections (no "Summary:", no headings). No code blocks unless the change is explicitly about code snippets.

## STEP-BY-STEP PROCEDURE

### 1. Understand the change

- Identify what changed (files/modules) and why (bugfix, feature, refactor, etc.).
- Extract the single most important intent of the commit.
- If there are multiple unrelated intents, pick the dominant one and keep the rest in body bullets (do not invent changes).

### 2. Choose subject style

Pick **exactly one** style:

**Style A — Conventional Commits (preferred by default):**
<type>(optional scope): <subject>

Use this style unless you have a clear reason not to.

**Style B — Plain imperative subject (only if requested or repo does not use Conventional Commits):**
<Subject>

Do not mix styles.

### 3. Write the subject line

#### Subject line rules (both styles):

- Use imperative present (e.g., Add, Fix, Refactor, Update, Remove, Rename, Prevent, Improve).
- No trailing period.
- Be specific; avoid vague words (e.g., "changes", "stuff", "updates" without object).
- Prefer length <= 50 characters; hard limit <= 72 characters.
- Mention what changed, not implementation details.

#### Style A rules (Conventional Commits):

- Allowed types (lowercase only):
  `feat, fix, refactor, perf, docs, test, build, ci, chore, style, revert`

- Optional scope:
    - Use scope only when it clearly helps (module/package/area), e.g., schedule, auth, db, api, ui.
    - Keep scope short (1–2 words), lowercase, no spaces if possible (use "-" if needed).

- Format:
  `type(scope): subject` OR `type: subject`

#### BREAKING CHANGES RULE ("!" AND DESCRIPTION)

If the commit introduces a breaking change (public API/contract/config behavior change that requires consumer updates):

- Prefer Style A and mark it with "!" in the subject line:
    - type!: subject
    - type(scope)!: subject

- In the body, include a dedicated bullet that starts with "BREAKING CHANGE:" and clearly describes:
    - what broke,
    - what consumers must change,
    - (optionally) migration notes.

- Example breaking bullet:
    - BREAKING CHANGE: Rename "userId" to "accountId" in API responses; update clients accordingly.

#### Style B rules (Plain):

- Capitalize the first letter of the first word.
- No prefix, no scope.
- If Style B must be used for a breaking change, start the subject with "BREAKING:" and still include a body bullet starting with "BREAKING CHANGE:".

### 4. Write the body (bullet list)

- Body is a bullet list; each bullet:
    - starts with "- "
    - describes one concrete change
    - uses imperative present
- Use as many bullets as needed, but avoid noise:
    - combine tiny changes if they belong together
    - keep unrelated changes separate
- Wrap lines at ~72 characters (soft rule).

- Do not repeat the subject verbatim; add details.

### 5. Forbidden content

Do not include:

- Issue numbers unless explicitly present in the diff/message policy.
- "WIP", "temp", jokes, emojis, excessive punctuation.
- Unverifiable claims (e.g., "improves performance by 50%") unless evidence is clearly in the diff.

## DECISION RULES (TYPE SELECTION)

Use these heuristics:

- feat: introduces new user-visible behavior or capability
- fix: corrects a bug, wrong behavior, edge case, crash, regression
- refactor: internal restructuring without behavior change
- perf: performance improvement (measurable or clearly intended)
- docs: documentation-only changes
- test: adds/updates tests only
- build: build system/dependencies/tooling (bundlers, compilers)
- ci: CI pipeline/workflows/scripts
- chore: maintenance tasks not affecting runtime behavior (cleanup, configs)
- style: formatting only (no logic change)
- revert: reverts a previous commit

If unsure between chore and refactor:

- Choose refactor if code structure changes meaningfully.
- Choose chore if it’s mostly housekeeping/config changes.

## TEMPLATES (COPY-READY)

Style A (Conventional Commits)
type(scope): subject

- Bullet 1
- Bullet 2
- Bullet 3

Style A without scope
type: subject

- Bullet 1
- Bullet 2

Style A with breaking change
type(scope)!: subject

- BREAKING CHANGE: <what broke and what to update>
- Bullet 2
- Bullet 3

Style B (Plain imperative)
Subject

- Bullet 1
- Bullet 2

Style B breaking (only if Style A is not allowed)
BREAKING: Subject

- BREAKING CHANGE: <what broke and what to update>
- Bullet 2

EXAMPLES

Example 1 — Conventional, refactor
refactor(schedule): restructure module layout

- Remove obsolete "old-schedule" directory
- Move services into "modules/schedule/services"
- Delete unused mock data

Example 2 — Conventional, fix
fix(api): prevent null access in user mapping

- Add guard for missing profile fields
- Return default values for optional properties

Example 3 — Conventional, breaking change
feat(api)!: change auth token format

- BREAKING CHANGE: Token is now JWT; update clients to send "Authorization: Bearer <token>"
- Update token validation middleware
- Adjust integration tests for new header format

Example 4 — Plain imperative
Refactor import statements for clarity

- Reorder imports in context.ts
- Update type filters in ctx.mapping.ts for BotContext
